# Feature: Post-Session Feedback Report

> Status: Planned. Current production flow returns final score text from `POST /api/chat` at session completion. Dedicated `/api/feedback` + report page flow below is target design, not current implementation.

## Purpose

The debrief that users receive after completing an interview session. This is where all evaluation and scoring happens — explicitly deferred from the interview itself. The report gives users actionable insight into how they performed, where they struggled, and what to work on before their next session.

This feature is the primary value delivery of the product. A good report makes users want to practice again.

**Users:** Authenticated (report belongs to the session owner)

---

## Routes

| URL | Page | Auth required |
|-----|------|--------------|
| `/report/[sessionId]` | Post-session feedback report | Yes (owner only) |

The report page sits outside the `(app)` route group intentionally — it can be made shareable (no auth wall) in the future without restructuring routes.

---

## Report Generation

The report is generated by the backend when the user ends a session.

**Trigger (planned):** User clicks "End Interview" on session page → `POST /api/feedback`.

**LLM prompt context sent to backend:**
- Full Q&A transcript (all `session_messages` in order)
- `job_role` and `question_type` from the session config

**Backend returns (`FeedbackReport`):**

```typescript
interface FeedbackReport {
  overall_rating: number          // 1–10
  clarity_score: number           // 1–10 — how clearly ideas were communicated
  structure_score: number         // 1–10 — logical flow and organization of answers
  relevance_score: number         // 1–10 — how well answers addressed the questions
  confidence_score: number        // 1–10 — inferred from vocabulary and directness
  suggestions: string[]           // 3–5 specific, actionable improvement tips
  transcript_summary: string      // 2–3 sentence AI-generated session summary
}
```

The report is saved to `feedback_reports` in Supabase immediately after generation.

## Report Page Layout

### 1. Header
- "Interview Complete" headline
- Session metadata: job role, question type, difficulty, date, duration
- "Back to Dashboard" link

### 2. Overall Score Card
- Large overall rating display (e.g. `7 / 10`)
- Visual indicator (progress ring or bar)
- One-line characterization based on score range:
  - 1–3: "Needs significant work"
  - 4–5: "Developing"
  - 6–7: "Good foundation"
  - 8–9: "Strong performance"
  - 10: "Exceptional"

### 3. Dimension Scores
Four score cards displayed in a 2×2 grid (or horizontal row on desktop):

| Dimension | What it measures |
|---|---|
| **Clarity** | How clearly and concisely ideas were communicated |
| **Structure** | Logical flow and organization of answers |
| **Relevance** | How directly answers addressed the questions asked |
| **Confidence** | Directness, assertiveness, and use of hedging language |

Each card shows: dimension name, score (X/10), and a short 1-line LLM-generated note (future enhancement).

### 4. AI Suggestions
Bulleted list of 3–5 specific, actionable improvement tips from the LLM.

Format: Each suggestion should be specific to this session — not generic tips. E.g.:
- "Use the STAR method when answering behavioral questions — your answer to 'difficult team member' lacked a clear outcome."
- Not: "Practice more."

### 5. Session Summary
1–2 paragraph narrative summary of the session generated by the LLM.

### 6. Full Transcript
Collapsible section (collapsed by default) showing the full Q&A exchange:
- AI questions in one style (e.g. muted card, left-aligned)
- User answers in another (e.g. blue card, right-aligned — consistent with interview page)

### 7. Actions
- "Practice Again" → `/interview/setup` (starts a new session)
- "Back to Dashboard" → `/dashboard`
- (Future) "Export as PDF" or "Share Report"

## Backend Calls

| Action | Endpoint | When |
|--------|----------|------|
| Generate report | `POST /api/feedback` | Triggered by "End Interview" on session page |

No backend calls are made when viewing the report page — data is loaded from Supabase directly.

---

## Data Loading (Report Page)

```
1. Get sessionId from URL params
2. Supabase: SELECT from interview_sessions WHERE id = sessionId AND user_id = auth.uid()
   → If not found or wrong user: redirect to /dashboard
3. Supabase: SELECT from feedback_reports WHERE session_id = sessionId
   → If not found: show error state ("Report could not be generated. Please try again.")
4. Supabase: SELECT from session_messages WHERE session_id = sessionId ORDER BY turn_index
5. Render report with all data
```

## Key Edge Cases

- **Report generation failed (LLM error):** If `POST /api/feedback` returns an error, navigate to the report page anyway and show a clear error state: "We couldn't generate your report this time. Your session has been saved." Include a "Try generating report" retry button that re-calls the feedback endpoint.
- **User navigates to report before it's generated:** If the report page loads before the `feedback_reports` INSERT has completed, show a loading skeleton. Poll or use a Supabase real-time subscription to detect when the row appears.
- **Very short session (1–2 rounds):** The LLM may produce a low-quality report with minimal data. Show a notice: "This report is based on a short session — practice with more rounds for more detailed feedback."
- **User tries to access another user's report:** RLS will block the Supabase query. The page receives an empty result and should redirect to `/dashboard` — do not render an error that reveals the session exists.
